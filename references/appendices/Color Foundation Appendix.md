## Current Scope

Color is a foundation-level system. It defines the palette, theme behavior, product identity mapping, semantic UI usage tokens, and documentation for humans and AI.

In scope now:

- Foundation color variables.
- Light and Dark theme behavior.
- B2C and B2B product modes.
- Semantic color tokens for foundation-level UI usage.
- Designer-assigned opacity modifiers for transparent focus/tint/scrim and overlay layers.
- Color documentation boards.
- AI guidance for generating designs with existing Dotty color tokens.

Out of scope now:

- Component color wiring.
- Component token mappings.
- Hover, focus, selected, loading, disabled, CTA, and other interaction-state decisions.
- Full overlay recipe tokenization and color-plus-opacity combination tokens.
- Gradient tokenization.
- Accessibility color-value corrections beyond documented caveats.

## Canonical Model

```txt
color.palette.* -> theme.palette.* + product.* -> color.*
```

The current Figma collection model is:

```txt
01 Color / Primitives -> 01 Color / Theme + 01 Color / Product -> 01 Color / Semantic
```

Primitives are raw values. Theme handles Light/Dark. Product handles product identity such as B2C/B2B. Semantic is the stable usage layer for product UI.

## Naming Contract

Canonical token IDs use dot notation in docs, code syntax, exports, and AI output.

Examples:

- `color.palette.coral.500`
- `theme.palette.coral.500`
- `product.brand.500`
- `color.background.surface`
- `color.text.default`

Figma variable names may still appear slash-grouped in the Figma UI because Figma rejects dots inside `Variable.name`. Those slash names are picker organization only. The `WEB` code syntax should carry the canonical dot token ID. CSS custom properties can be generated downstream by Style Dictionary or equivalent tooling.

## Primitive Layer

`01 Color / Primitives` contains:

- Light/source ramps: `coral`, `navyBlue`, `grey`, `lightBlue`, `red`, `green`, and `orange`.
- Dark ramps under `color.palette.dark.*`.
- Common values: `color.common.white`, `color.common.black`, `color.common.transparent`.

Current important values:

| Token | Value | Note |
| --- | --- | --- |
| `color.palette.coral.500` | `#FA6B6B` | Brand coral. |
| `color.palette.neutral.navyBlue.500` | `#132968` | Core navy text and structure color. |
| `color.palette.neutral.grey.50` | `#F5F6F8` | Default quiet background layer. |
| `color.palette.dark.red.500` | `#F0445F` | Dark error, intentionally cooler than coral. |
| `color.palette.dark.coral.500` | `#FF7979` | Dark brand coral. |

Dark mode is not generated by simple inversion. Dark primitives are tuned from the light ramps for dark surfaces, preserving hue family while adjusting lightness and chroma.

## Theme Layer

`01 Color / Theme` owns appearance behavior.

Examples:

- `theme.palette.coral.500`: Light resolves to `color.palette.coral.500`; Dark resolves to `color.palette.dark.coral.500`.
- `theme.common.surface`: Light resolves to `color.common.white`; Dark resolves to `color.palette.dark.neutral.surface`.
- `theme.background.strong`: Light resolves to `color.palette.neutral.navyBlue.500`; Dark resolves to `color.palette.dark.neutral.navyBlue.100`.

Do not add usage roles such as text, border, card, or status to Theme. Theme is a mode-ready palette layer.

## Product Layer

`01 Color / Product` owns product identity.

Current groups:

- `product.brand.{50..900}`
- `product.neutral.navyBlue.{50..900}`
- `product.neutral.grey.{50..900}`
- `product.information.{50..900}`
- `product.success.{50..900}`
- `product.warning.{50..900}`
- `product.error.{50..900}`
- `product.common.white`
- `product.common.black`
- `product.common.transparent`
- `product.common.surface`
- `product.background.strong`

B2C and B2B currently resolve to the same Theme values. Product-specific differences should happen in Product, not by forking Semantic usage names.

## Semantic Layer

`01 Color / Semantic` is the current usable UI layer. It has one `Value` mode. Do not add a Dark mode here.

### Text Tokens

- `color.text.default`
- `color.text.subtle`
- `color.text.muted`
- `color.text.disabled`
- `color.text.inverse`
- `color.text.brand`
- `color.text.information`
- `color.text.success`
- `color.text.warning`
- `color.text.error`

Generic text intentionally uses the navyBlue ramp. `color.text.inverse` resolves to white for brand, dark, or high-contrast surfaces.

### Icon Tokens

- `color.icon.default`
- `color.icon.subtle`
- `color.icon.muted`
- `color.icon.disabled`
- `color.icon.inverse`
- `color.icon.brand`
- `color.icon.information`
- `color.icon.success`
- `color.icon.warning`
- `color.icon.error`

Icon tokens alias directly to Product values. Do not alias icon tokens to text tokens.

### Background Tokens

- `color.background.default`
- `color.background.surface`
- `color.background.subtle`
- `color.background.brand`
- `color.background.brand.subtle`
- `color.background.strong`
- `color.background.information`
- `color.background.success`
- `color.background.warning`
- `color.background.error`

`color.background.surface` is the default card/container fill. In Light it resolves to white; in Dark it resolves to the dark neutral surface primitive. White is treated as a common surface value, not as part of the neutral color family.

`color.background.strong` is for small high-emphasis navy surfaces and should be paired with `color.text.inverse` and `color.icon.inverse`.

### Border Tokens

- `color.border.default`
- `color.border.subtle`
- `color.border.strong`
- `color.border.information`
- `color.border.success`
- `color.border.warning`
- `color.border.error`

Generic border tokens are neutral structural tokens. Status borders are supportive signals and should not be the only indication of status.

## Opacity Modifiers

`01 Color / Opacity` is separate from the color architecture. It stores approved opacity percentages as FLOAT variables scoped to opacity, not as color tokens. In Figma, these values are stored as percentage numbers (`16` for `opacity.16`), not normalized alpha decimals (`0.16`); export tooling can normalize them downstream when a code target expects alpha decimals.

| Token | Value | Intended use |
| --- | ---: | --- |
| `opacity.8` | 8% | Very subtle tint or overlay layer when intentionally assigned by a designer. |
| `opacity.12` | 12% | Low focus halo or tint layer. |
| `opacity.16` | 16% | Standard focus halo or tint layer. |
| `opacity.24` | 24% | Stronger focus halo or tint layer. |
| `opacity.40` | 40% | Default modal/page overlay scrim. |
| `opacity.80` | 80% | Strong blocking or processing overlay. |

Rules:

- Apply opacity as a separate layer using a base color token plus an opacity token.
- Do not apply opacity to whole UI groups because child text, icons, focus indicators, and controls must stay independently tokenized and legible.
- Do not invent arbitrary opacity values.
- Do not create color-plus-opacity combination tokens unless a stable semantic recipe is intentionally approved.
- AI must not introduce opacity in generated designs unless explicitly requested or preserving an existing designer-assigned treatment.

## Role Guidance

Neutral is one role with two family jobs. Navy blue carries readable content, default icons, hierarchy, and key values. Grey carries backgrounds, dividers, disabled states, separators, and quiet structure.

| Role | Use for | Do not use for |
| --- | --- | --- |
| Brand / coral | Omio identity, approved primary brand emphasis, promotional emphasis, small active indicators, restrained brand anchors when identity is needed, and one consistent Omio-branded prototype action emphasis when the final action spec is unresolved. | Error, destructive, essential small text, generic status meaning, generic labels, pretitles, selected-row/card washes, or mixed competing action styles. |
| Neutral / navyBlue | Primary reading color, hierarchy, strong structure, icons. | Generic decoration or quiet dividers. |
| Neutral / grey | Page backgrounds, disabled states, separators, quiet structure. | Primary text or status meaning. |
| Information / lightBlue | Guidance, selected-support surfaces, neutral information. | Brand identity or success/error messages. |
| Success | Positive outcome, completed state, valid outcome. | Generic decoration. |
| Warning | Caution, risk, reversible issue before error. | Failed/destructive state. |
| Error | Failed, invalid, destructive, serious feedback. | Brand coral or marketing emphasis. |
| Inverse | Readable text/icons on brand, dark, or high-contrast fills. | A replacement for dark-mode theming. |
| Accent / feature | Expressive feature color systems such as AI or Omio Pass. | Core status, brand, or interaction-state defaults. |

Gradients and accent feature palettes are documentation references only for now. Each feature should own its source documentation and link back to the color docs so designers, engineers, and AI know the accent exists.

## Adding A Color

New color creation is an exception workflow. Designers or AI agents should not add a new color during normal design generation.

When a new color is explicitly requested, follow `Color Docs / 05 Adding a color`:

1. Define the purpose and whether it is a flat accent, gradient accent, or new universal role.
2. Create or document the primitive source values.
3. Create Light and Dark ramp behavior if the color enters the token system.
4. Route through Theme.
5. Route through Product if the color is product-owned.
6. Expose Semantic only if there is a stable reusable UI meaning.
7. Document usage, accessibility, product behavior, and caveats.

## Usage Contract

When Dotty colors are used by designers, engineers, or AI agents:

- Use existing semantic tokens first.
- Never invent raw hex if a token exists.
- Never add a new color, ramp, gradient, accent, semantic token, or raw hex unless the prompt explicitly asks to create or extend the color system.
- Never add opacity to generated UI unless the prompt explicitly asks for opacity or the AI is preserving an existing designer-assigned opacity treatment.
- If opacity is requested, use only `opacity.8`, `opacity.12`, `opacity.16`, `opacity.24`, `opacity.40`, or `opacity.80`, and describe it as base color token plus opacity token on a separate layer.
- Use Product only for product identity decisions.
- Use Theme only to explain appearance behavior.
- Use Primitives only when explaining source values or creating approved ramps.
- Do not add Dark mode to Semantic.
- Do not use coral as error.
- Do not infer hover, focus, selected, loading, disabled, CTA, or other interaction-state styling from color foundations alone; state the unresolved source-spec dependency.
- Do not use warning, error, or success backgrounds to invent selected states unless that status is the actual meaning.
- Keep Omio-branded prototype primary action emphasis on the brand role; if final action styling is not documented, state that source-spec dependency.

Required AI color output:

- Chosen token.
- Reason.
- Mode behavior.
- Product behavior.
- Accessibility note or caveat.
- Whether a new color was proposed.

## Online Docs Notes

The online docs should expose:

- Overview of the color model.
- Roles and role guidance.
- Token layers and token selection order.
- Practical examples of applying tokens to UI.
- Opacity modifiers and usage rules.
- How to add a color.
- Usage rules.

Keep the online docs tool-neutral. Figma is one implementation view; the same token IDs should work for designers, engineers, docs, and AI.
